    class Quark::Mdk::Session

    n A session provides a lightweight sequential context that an microservice can use in the context of any applicationn framework in order to manage its interactions with othern microservices. It provides simple APIs for servicen resolution, distributed tracing, and circuit breakers.nn A microservices architecture enables small self containedn units of business logic to be implemented by separate teamsn working on isolated services based on the languages andn frameworks best suited for their problem domain.nn Any given microservice will contain sequential business logicn implemented in a variety of ways depending on the applicationn framework chosen. For example it may be a long runningn thread, a simple blocking request handler, or a chainedn series of reactive handlers in an async environment.nn For the most part this business logic can be implementedn exactly as prescribed by the application framework of choice,n however in a microservices architecture, some special caren needs to be taken when this business logic interacts withn other microservices.nn Because microservices are updated with much higher frequencyn than normal web applications, the interactions between themn form key points that require extra care beyond normal webn interactions in order to avoid creating a system that is bothn extremely fragile, unreliable, and opaque.nn Realtime service resolution, distributed tracing, andn resilience heuristics such as circuit breakers provide then foundational behavior required at these interactionn points. These capabilites must be combined with the defensiven coding practice of intelligent fallback behavior when remoten services are unavailable or misbehaving, in order to build an robust microservice application.nn Because of this, a session is expected to be created and maden available to all business logic within a given microservice,n e.g. on a per request basis, as a thread local, part of an context object, etc depending on the application framework ofn choice.n
