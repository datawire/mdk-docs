MDK

\n The MDK API consists of two interfaces: MDK and Session. The\n MDK interface holds globally scoped APIs and state associated\n with the microservice. The Session interface holds locally\n scoped APIs and state. A Session must be used sequentially.\n\n The MDK instance is responsible for communicating with\n foundational services like discovery and tracing.\n\n There will typically be one MDK instance for the entire\n process, and one instance of the Session object per\n thread/channel/request depending on how the MDK is integrated\n and used within the application framework of choice.\n

Session

\n A session provides a lightweight sequential context that a\n microservice can use in the context of any application\n framework in order to manage its interactions with other\n microservices. It provides simple APIs for service\n resolution, distributed tracing, and circuit breakers.\n\n A microservices architecture enables small self contained\n units of business logic to be implemented by separate teams\n working on isolated services based on the languages and\n frameworks best suited for their problem domain.\n\n Any given microservice will contain sequential business logic\n implemented in a variety of ways depending on the application\n framework chosen. For example it may be a long running\n thread, a simple blocking request handler, or a chained\n series of reactive handlers in an async environment.\n\n For the most part this business logic can be implemented\n exactly as prescribed by the application framework of choice,\n however in a microservices architecture, some special care\n needs to be taken when this business logic interacts with\n other microservices.\n\n Because microservices are updated with much higher frequency\n than normal web applications, the interactions between them\n form key points that require extra care beyond normal web\n interactions in order to avoid creating a system that is both\n extremely fragile, unreliable, and opaque.\n\n Realtime service resolution, distributed tracing, and\n resilience heuristics such as circuit breakers provide the\n foundational behavior required at these interaction\n points. These capabilites must be combined with the defensive\n coding practice of intelligent fallback behavior when remote\n services are unavailable or misbehaving, in order to build a\n robust microservice application.\n\n Because of this, a session is expected to be created and made\n available to all business logic within a given microservice,\n e.g. on a per request basis, as a thread local, part of a\n context object, etc depending on the application framework of\n choice.\n

start_interaction ()

\n Start an interaction with a remote service.\n\n The session tracks any nodes resolved during an\n interactin with a remote service.\n\n The service resolution API permits a compatible instance\n of the service to be located. In addition, it tracks\n which exact instances are in use during any\n interaction. Should the interaction fail, circuit breaker\n state is updated for those nodes, and all involved\n instances involved are reported to the tracing services.\n\n This permits realtime reporting of integration issues\n when services are updated, and also allows circuit\n breakers to mitigate the impact of any such issues.\n

start_interaction()
